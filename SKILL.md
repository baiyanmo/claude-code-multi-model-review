# 多模型代码审查

调用用户已配置的多个外部模型并行审查当前 git diff，各模型给出独立意见，最后合成综合报告。

## 工作流程

### 步骤 1：检查模型库

读取 skill 所在目录下的 `models.json`。

**如果 models 为空 → 引导用户首次配置：**

询问用户要添加的模型信息：
1. 给模型起个名字（如 "豆包"）
2. API Base URL（OpenAI 兼容格式，如 `https://ark.cn-beijing.volces.com/api/v3`）
3. API Key
4. 模型 ID（如 `deepseek-v4-pro`、`ep-xxx`）

然后将配置写入 models.json。建议至少配 2 个模型。配完后继续步骤 2。

### 步骤 2：选择本次使用的模型

列出 models.json 中所有已配置的模型，问用户本次审查要用哪几个（可多选）。

如果只有一个模型，直接用它（退化为单模型审查，但仍保留扩展能力）。

**可选：** 问用户是否要给某个模型指定审查侧重点（如"重点关注安全问题"），该侧重点会传 `--angle` 给 review.js。

### 步骤 3：获取 diff

```bash
git -C <项目根目录> diff HEAD
```

如果返回空，改用：

```bash
git -C <项目根目录> diff --cached
```

如果还是空，告知用户没有改动，结束。

如果 diff 超过 3000 行，提醒用户 diff 较大，是否需要限制范围。

### 步骤 4：并行审查

对于用户选中的每个模型：

```bash
node review.js \
  --diff <diff临时文件路径> \
  --model <模型key> \
  --config models.json \
  [--angle "<审查侧重点>"]
```

**关键：** 所有模型的后台任务同时启动，不要串行等待。

用 `run_in_background: true` 并行启动，等待全部完成后收集结果。

### 步骤 5：合成报告

将所有模型的返回收集齐后，输出综合报告。报告结构：

```
# 多模型代码审查报告

## 参与模型
| 模型 | 厂商 | 模型 ID |
|------|------|---------|

## 各模型审查意见
（每个模型一栏，保留其原始输出）

### 模型 A
...

### 模型 B
...

## 交叉验证
（列出多个模型同时发现的问题 — 置信度最高）
- 模型A、C 同时指出: ...

## 分歧对比
（列出不同模型给出不同意见的地方）
| 问题 | 模型A | 模型B |

## 总结
（综合各模型意见，给出你认为最值得关注的问题）
```

不要删减模型的原始输出。完整保留每个模型的意见。

## 配置模型

用户可以在任何时候说"添加审查模型"来增加新模型。流程同步骤 1 的首次配置。

说"管理审查模型"可以查看/删除已配置的模型。

## 注意事项

- models.json 包含 API Key，不要提交到 git
- review.js 依赖 Node.js 内置模块（fs, path, https, http），不需要 npm install
- API 超时设为 120 秒，diff 过大时可能超时
